Возможность объединения: эффекты 2 + 3

Оба подписываются на document events при isOpen и делают cleanup. Два отдельных прохода commit-phase при открытии.

Нюанс: эффект 2 зависит от [isOpen, close, triggerRef], эффект 3 — от [isOpen, close, triggerRef, overlayNode]. При объединении Escape-handler будет перерегистрироваться при появлении overlayNode. Это допустимо: overlayNode меняется только один раз (null → элемент) при открытии. 
// Объединённый эффект: Escape + click outside
useEffect(() => {
    if (!isOpen) return;

    function handleKeyDown(event: KeyboardEvent) {
        if (event.key === 'Escape') {
            close();
            triggerRef.current?.focus();
        }
    }

    function handleClick(event: MouseEvent) {
        const target = event.target as Node;
        const trigger = triggerRef.current;
        if (trigger?.contains(target) || overlayNode?.contains(target)) return;
        close();
    }

    document.addEventListener('keydown', handleKeyDown);
    const timeoutId = window.setTimeout(() => {
        document.addEventListener('click', handleClick);
    }, 0);

    return () => {
        document.removeEventListener('keydown', handleKeyDown);
        clearTimeout(timeoutId);
        document.removeEventListener('click', handleClick);
    };
}, [isOpen, close, triggerRef, overlayNode]);

2. Рендер-цикл при открытии

При клике на триггер — каскад state → effect → state:

Клик → toggle() → setIsOpen(true) ← рендер 1
  эффект 4: mount() → setOverlays() в OverlayProvider ← рендер 2 (Provider + все children)
    React рендерит портал, ref callback → setOverlayNode(div) ← рендер 3 (потребитель)
      эффект 3 (click outside): теперь видит overlayNode
      эффект 5 (autoUpdate): стартует

Итого: 3 рендера при открытии.

Проблема в OverlayProvider: context value {{ mount, unmount }} — новый объект при каждом рендере провайдера. Хотя mount и unmount обёрнуты в useCallback, сам объект пересоздаётся → все потребители OverlayContext перерендериваются, даже если их overlay не изменился.

Решение: мемоизация context value

// OverlayProvider.tsx
const contextValue = useMemo(() => ({ mount, unmount }), [mount, unmount]);

return (
    <OverlayContext.Provider value={contextValue}>
        {children}
        {createPortal(/* ... */, document.body)}
    </OverlayContext.Provider>
);

3. Event listeners

setTimeout(0) для click outside (строка 96)

Без него клик по триггеру, который открывает попап, тут же ловится только что подписанным listener-ом и закрывает попап. setTimeout(0) откладывает подписку до следующего макротаска.

Альтернатива: mousedown вместо click

mousedown для нового клика не может совпасть с текущим click-event. Это стандартный паттерн в Radix, Headless UI и других библиотеках. Убирает setTimeout, убирает гонку с другими обработчиками.

// Без setTimeout — сразу подписываемся
document.addEventListener('mousedown', handleClickOutside);

Недостаток: закрытие при нажатии, а не отпускании — менее привычно, но общепринятое поведение для dismissal.

Утечки

Утечек нет, cleanup корректный. clearTimeout + removeEventListener вызываются в правильном порядке.

4. autoUpdate из @floating-ui/react

@floating-ui/react (~34 KB minified) используется в проекте ровно в одном месте — ради функции autoUpdate.

Что делает autoUpdate (по дефолту)

МеханизмВключён
ResizeObserver на обоих элементахДа
scroll на ancestor-контейнерахДа
resize на windowДа
animationFrame pollingНет (опция)

Замена: собственная реализация

Важно: ранее собственная реализация (useOverlayPosition) была медленной. Причина — не механизм подписки, а setState в обработчике scroll (каждый scroll event → React рендер → virtual DOM diff). В usePopover используется прямая мутация DOM через applyTriggerMirror — ноль рендеров React, поэтому замена безопасна. Необходимо добавить rAF-батчинг для scroll/resize handlers, т.к. scroll-событие может стрелять многократно за один кадр. 
useEffect(() => {
    const trigger = triggerRef.current;
    if (!isOpen || !trigger || !overlayNode) return;

    applyTriggerMirror(trigger, overlayNode); // начальная позиция

    const ro = new ResizeObserver(() => {
        applyTriggerMirror(trigger, overlayNode);
    });
    ro.observe(trigger);
    ro.observe(overlayNode);

    let rafId: number | null = null;

    function handleUpdate() {
        if (rafId !== null) return;
        rafId = requestAnimationFrame(() => {
            rafId = null;
            applyTriggerMirror(trigger!, overlayNode!);
        });
    }

    window.addEventListener('scroll', handleUpdate, true);
    window.addEventListener('resize', handleUpdate);

    return () => {
        if (rafId !== null) cancelAnimationFrame(rafId);
        ro.disconnect();
        window.removeEventListener('scroll', handleUpdate, true);
        window.removeEventListener('resize', handleUpdate);
    };
}, [isOpen, triggerRef, overlayNode]);

Покрытие идентично autoUpdate (без animationFrame polling, который отключён по дефолту). Результат: −34 KB бандла.

5. mount/unmount через контекст

Двойной unmount при закрытии

Когда isOpen переключается true → false:

  1. Тело эффекта с isOpen === false: вызывает unmount(overlayId)
  2. Cleanup этого эффекта: return () => unmount(overlayId) — ещё один unmount

Map.delete идемпотентен, так что это не баг, но лишний setOverlays → лишний рендер провайдера.

Решение: unmount только в cleanup

useEffect(() => {
    if (!isOpen) return;

    mount(
        overlayId,
        <div ref={setOverlayNode} style={{ /* debug */ }}>
            {content({ id: popoverId, placement, onClose: close })}
        </div>,
    );

    return () => unmount(overlayId);
}, [isOpen, overlayId, popoverId, placement, content, close, mount, unmount]);

6. Стабильность ссылок: content

content — функция (props: PopoverRenderProps) => ReactNode, передаваемая потребителем. Типичное использование:

const { triggerProps } = usePopover(ref, {
    content: ({ id, placement, onClose }) => (
        <Popover id={id} placement={placement}>...</Popover>
    ),
});

Стрелочная функция пересоздаётся при каждом рендере. Она входит в зависимости эффекта 4. При isOpen === true каждый рендер потребителя (от любой причины):

  1. Перезапускает эффект 4
  2. Вызывает mount() → setOverlays
  3. Ререндерит провайдер
Это главная проблема производительности хука.

Вариант A: ref на content, компонент-обёртка

const contentRef = useRef(content);
contentRef.current = content;

useEffect(() => {
    if (!isOpen) return;

    mount(
        overlayId,
        <PopoverPortal
            contentRef={contentRef}
            popoverId={popoverId}
            placement={placement}
            onClose={close}
            onOverlayRef={setOverlayNode}
        />,
    );

    return () => unmount(overlayId);
}, [isOpen, overlayId, popoverId, placement, close, mount, unmount]);
// ^^^ content убран из зависимостей

Подвох: если потребитель обновил content (изменил текст), PopoverPortal об этом не узнает — он рендерится только при mount().

Вариант B: изменить API — хук возвращает props, потребитель рендерит сам

const { triggerProps, overlayProps, isOpen } = usePopover(ref, { placement });

// В JSX потребителя:
{isOpen && (
    <Portal>
        <div {...overlayProps}>
            <Popover placement={placement}>...</Popover>
        </div>
    </Portal>
)}

Устраняет проблему полностью — хук не знает о содержимом. Но это смена API.

Вариант C (минимальный): dev-warning + документация useCallback

// Рекомендация потребителю:
const content = useCallback(
    ({ id, placement, onClose }: PopoverRenderProps) => (
        <Popover id={id} placement={placement}>...</Popover>
    ),
    [],
);
